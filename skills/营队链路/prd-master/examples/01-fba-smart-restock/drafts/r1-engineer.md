## §5 详细功能说明

### 5.1 通用约定（时区 / 格式 / 货币 / i18n / 命名）

| 项 | 约定 | 说明 |
|---|------|------|
| 时区 | 所有持久化时间用 UTC（ISO 8601, 含毫秒+Z） | 解决 R1-T8 |
| 业务日切 | 按"美西站点本地时间"日切（PST/PDT，含夏令时） | 销量/库存以亚马逊站点为准；其他站点扩展时单独配置 |
| 展示时区 | UI 默认 Asia/Shanghai，用户可切换 | 顶部时区指示器 |
| 货币 | 原值以销售站点币种存储（USD 为主）；展示按"原币 + 折算 CNY"双显示 | 汇率每日 9:00 UTC 拉取，缓存 24h |
| 数量精度 | 件 = 整数；金额 = 2 位小数；天数 = 1 位小数；百分比 = 1 位小数 | 所有 UI/导出一致 |
| SKU 编号 | 内部 SKU ID = UUIDv4；展示用 seller-sku（卖家自定义字符串，≤ 40 字符） | 解耦 ASIN 与 SKU |
| 时间窗口 | 决策计算窗口 T = 90 天滚动；历史训练窗口 = 365 天（不足按可得长度） | |
| 空值规则 | 必填缺失 → 拒绝写入并标 error；可选缺失 → null + 标 warning | |
| i18n | v1 仅中文（zh-CN）；预留 i18n key，不硬编码字符串 | v1.5 可加英文 |
| 文件导出 | CSV 用 UTF-8 BOM（Excel 兼容）；时间列附 ISO 字符串列 | |
| ID 幂等 | 所有创建型 API 必须接受 client_request_id（幂等键，TTL 24h） | 解决重复提交 |
| 取整规则 | 数量上取整（ceil），金额四舍五入（half-even） | |

### 5.2 FR-1 数据接入 Adapter

**位置**: 系统底座 / 后台模块"数据源管理"页 + "数据快照"页

**界面元素与展示规则**:

| 元素 | 类型 | 默认态 | 操作后 | 禁用条件 |
|---|---|---|---|---|
| 数据源列表（12+4 项 D1~D16） | Table | 显示名称/类型/最后同步/状态徽章（绿/黄/红） | 点击进详情 | 用户无 admin 权限时只读 |
| 模式切换 Toggle | Switch | "Mock" 模式（v1 默认） | 切到 Real 模式弹二次确认；切换记录 audit log | Real 数据源未完成接入时禁用 Real 档 |
| 同步频率选择 | Dropdown | 日（凌晨 02:00 UTC） | 可选 实时/15min/1h/日；改动需 admin 确认 | 数据源不支持 push 时实时档禁用 |
| 立即同步按钮 | Button | 可点 | 触发即时拉取，进度条显示 | 上次同步 < 60s 内禁用（防抖） |
| 数据质量校验报告 | Panel | 折叠 | 展开显示 缺失率/异常值数/重复键数 + 每项的 sample 5 行 | 无最近一次同步时禁用 |
| 数据快照列表 | Table | 倒序 30 天 | 选中可"复制为决策输入"用于回测 | 快照已被引用于在生效的决策时禁用删除 |
| Schema 字段映射器 | Form | v1 预填 FBA API 标准 schema | 修改保存后版本号 +1，旧版本仍可回滚 | 字段类型与下游约束不符时保存按钮禁用并红字提示 |

**交互逻辑**:
1. **Adapter 抽象层**：使用 Strategy 模式定义 `IDataAdapter`接口（`fetch(since, until) -> RawBatch`、`normalize(RawBatch) -> CanonicalBatch`、`validate(CanonicalBatch) -> ValidationReport`）；v1 提供 `MockAdapter` 与 `FBAStubAdapter`（按真实 FBA Reports schema 反向设计），v1.5 切换只换实现不动接口。
2. **数据写入**：每次同步形成不可变 snapshot（snapshot_id = uuid + sha256(payload)），写 append-only 表 `data_snapshots`；当前指针表 `current_snapshot_ref` 单条记录指向最新有效快照。
3. **统一标准时间**：每条 raw record 入库时附 `source_tz`、`source_ts`、`utc_ts`、`business_date_pst` 四个字段，全链路用 utc_ts 计算，展示按 business_date_pst 聚合。
4. **数据质量校验**：6 类规则——必填非空 / 类型校验 / 值域校验（如销量 ≥0、价格 >0）/ 引用完整性（SKU 存在）/ 异常值（z-score>4 标黄、>6 标红）/ 时序连续性（缺日填 0 标 warning）。
5. **Mock→Real 切换审计**：每次切换记录操作人/时间/前后模式/原因，写 `audit_log` 表；UI 顶部沙盘斑马条样式由模式决定（FR-10 联动）。

**异常场景**:

| 类型 | 场景 | 处理 |
|---|---|---|
| 网络异常 | 数据源 API 超时（>30s） | 重试 3 次（指数退避 1s/4s/16s）；仍失败 → 用上次快照 + 顶部红条提示 + 写 alert |
| 网络异常 | DNS 解析失败 | 立即降级到 last-known-good 快照；推送 P1 告警 |
| 数据异常 | 字段缺失 > 5% | 拒绝整批写入，保留上次快照；UI 弹错误详情 + 下载 sample 失败行 |
| 数据异常 | 销量出现负值/价格为 0 | 该 SKU 标 quarantine，不进入优化；通知用户人工处理 |
| 数据异常 | 同一 SKU 同一天两条不同库存 | 取 source_ts 最新；记 conflict log |
| 权限不足 | 用户切 Real 模式但无 admin | 弹"无权限"toast，按钮恢复 Mock 态 |
| 并发冲突 | 两个 admin 同时改 Schema 映射 | 乐观锁（version 字段），后保存方报 409 + 提示"已有更新，请刷新" |
| 第三方失败 | FBA API 返 5xx | 重试 + 降级；连续 24h 失败 → 升级为 P0 告警（推送给管理员） |
| 第三方失败 | API 限流 429 | 按 Retry-After 头退避；记 rate-limit metric |
| 设备差异 | 移动端查看时数据源列表过宽 | 横向滚动 + 关键列固定（名称/状态） |
| 用户操作冲突 | 删除快照时该快照正被回测引用 | 阻止删除 + 提示"被回测任务 X 引用，先取消任务" |

**边界数值**:
- 单源单次拉取上限：10 万行；超出则分页（page_size=1万，cursor 模式）
- 数据保留：原始快照 90 天，归档冷存（S3 Glacier）至 2 年
- 单 SKU 单日多条记录：保留最近 1 条（其余进 history 表）
- API 调用预算：单源 ≤ 1000 req/min（避免限流）
- snapshot 写入超时：单次 ≤ 120s

### 5.3 FR-2 销量预测（区间预测）

**位置**: 后台 / "预测中心" + 主 Dashboard 内嵌区间图

**界面元素与展示规则**:

| 元素 | 类型 | 默认态 | 操作后 | 禁用条件 |
|---|---|---|---|---|
| SKU 选择器 | Combo | 默认显示 Top 20 销量 SKU | 多选 + 搜索 + 标签筛选 | 数据源未初始化禁用 |
| 预测窗口 | Slider | 30 天 | 7/30/60/90 可选 | - |
| 预测曲线图 | Chart | 中位数实线 + p10/p90 阴影 + 历史虚线 | hover 显示当日数值与置信区间 | 训练数据 <14 天时只画历史 + 灰色"数据不足" |
| 模型版本 | Tag | 显示当前 model_id + 训练日期 | 点击进 模型详情/对比 | - |
| 重新训练按钮 | Button | 可点（需 admin） | 触发后台 job，进度条 | 上次训练 <6h 内禁用 |
| 特征贡献条 | Bar | 默认折叠 | 展开显示 销量历史/广告/BSR/季节因子 占比 | 模型不支持解释时灰显 |
| 置信度等级标签 | Badge | 高/中/低 | 根据 p90-p10 区间宽度自动 | - |

**交互逻辑**:
1. **模型方案**：分层路由 —— 稳定 SKU（FR-16 输出"稳定"类）用季节性朴素法 + Holt-Winters；波动 SKU 用 LightGBM 分位数回归（quantile 0.1 / 0.5 / 0.9 三个模型同训）；长尾 SKU 直接出"低需求标签 + 安全库存常数"。
2. **特征**：销量历史（滞后 1/7/14/28）/ 7 天均值与标准差 / DOW / 月内位次 / 广告 spend / BSR / 价格 / 是否促销日（外部日历）/ 是否缺货历史日。
3. **训练频率**：每日凌晨 01:00 UTC 增量训练（在线学习的实现：保留模型 + 用最新 30 天微调）；每周日全量重训。
4. **区间→优化引擎对接**：FR-2 输出 `{p10, p50, p90}`；FR-4 引擎在 §5.5 N2 解决方案中明确——"v1 用 p50 求最优 + 用 p10/p90 跑 robust 情景，三套解的差异作为决策稳健度指标"。
5. **置信度→人话翻译**（FR-6 联动）：区间宽度 / p50 ≤ 0.3 → 高；0.3–0.6 → 中；>0.6 → 低。

**异常场景**:

| 类型 | 场景 | 处理 |
|---|---|---|
| 数据异常 | SKU 历史 <14 天 | 走"新品冷启动"分支：用同品类均值 ×0.5；标注"低置信度，建议人工确认" |
| 数据异常 | 全 0 历史（从未售出） | 不出预测，进入"长尾"分类（FR-16），决策清单不展示 |
| 数据异常 | 单日异常峰值（>10× p99） | 检测为 outlier，训练前 winsorize（截断到 p99）；UI 标"已修正异常值" |
| 网络异常 | 训练 job 节点崩溃 | 通过 job 重启策略恢复；保留上一个稳定模型快照（不影响在线预测） |
| 并发冲突 | 重训中触发再训请求 | 队列化，第二个请求合并到当前（去重） |
| 第三方失败 | 外部日历（促销日）API 失败 | 该特征 fallback 为 0；模型权重不变 |
| 设备差异 | 移动端图表过窄 | 自动切换为"数值表"形式 |
| 用户操作冲突 | 用户回滚到旧模型时旧模型已被清理 | 弹错并列出可用版本 |

**边界数值**:
- 训练窗口下限：14 天（不足则只出 p50 简单滑动均值）
- 异常值截断：z-score > 6 winsorize
- 模型保留版本数：30 个（roll-off）
- 训练单 SKU 时长上限：30 s；超时记 timeout、用上版本预测
- 区间显示精度：销量单位件，p50 取整；区间下界 max(0, floor(p10))

### 5.4 FR-3 经济目标函数实现

**位置**: 后端核心服务"计算引擎 / 目标函数模块"（无独立 UI；通过 FR-6 "可解释面板"间接展示）

**界面元素与展示规则**（仅"参数管理"页 UI）:

| 元素 | 类型 | 默认态 | 操作后 | 禁用条件 |
|---|---|---|---|---|
| 参数列表（5 项 + 3 个全局） | Table | 显示当前值/默认值/单位/最后修改人 | 点编辑弹 Modal | 用户非 admin 只读 |
| 缺货排名恢复倍数 | NumberInput | 3.0（默认） | 区间 [1, 10]，步长 0.1；保存即生效（影响下次计算） | - |
| 年化资金成本率 | NumberInput | 10%（默认） | 区间 [0, 50]，步长 0.5% | - |
| FBA 费率表导入 | FileUpload | 显示当前生效版本日期 | 上传 CSV 替换；保存前自动 diff 显示新旧差异 | 文件格式错时禁用保存 |
| 参数变更历史 | Timeline | 默认最近 10 条 | 可回滚到任意历史版本 | 当前正在跑的决策任务关联的版本不可删 |

**交互逻辑**:
1. **目标函数实现**：采用插件式函数注册 —— 每项成本/收益作为一个 `CostComponent`（输入：决策变量 + 参数 + 预测；输出：标量值 + 梯度签名）；总目标 = Σ components；新增/替换组件不动求解器。
2. **缺货分段**：缺货损失分两段——day 0..14 线性（=日销售额）；day 15..∞ 加 `rank_recovery_multiplier × decay(d)` 项（decay = 指数衰减 30 天衰到 1）。
3. **存储**：所有参数版本化（`parameters` 表 with effective_from / effective_to），决策计算时绑定 snapshot 的参数版本号，确保事后可复现。
4. **单位一致性**：所有金额内部用 USD 微元（micro-USD = 1e-6 USD，整数）避免浮点误差。
5. **目标函数 self-test**：每次发布前跑 50 条预置 case（已知期望值的极简场景），失败则阻断发布。

**异常场景**:

| 类型 | 场景 | 处理 |
|---|---|---|
| 数据异常 | FBA 费率表 SKU 体积缺失 | 该 SKU 用品类平均费率，并标 warning |
| 数据异常 | 缺货倍数被改成负数 | 表单校验拦截；后端兜底 max(0.1, x) 并记 audit |
| 并发冲突 | 决策跑到一半参数被改 | 决策绑定参数 snapshot，本次任务不受影响；下次任务用新参数 |
| 权限不足 | 普通用户改参数 | 403 + UI 隐藏编辑按钮 |
| 第三方失败 | 汇率 API 失败 | 用最近 24h 缓存；超 24h 用 30 天均值并标 warning |
| 用户操作冲突 | 回滚参数到已被清理的版本 | 提示并阻止 |

**边界数值**:
- 参数版本保留：100 个 + 永久保留"被引用于已执行决策"的版本
- 单次目标函数评估 < 5 ms（200 SKU × 90 天 × 2 物流方式 = 36000 变量）
- 浮点容差：1e-6 USD

### 5.5 FR-4 MINLP 优化引擎

**位置**: 后端核心 / 调度任务（无独立 UI；通过 FR-6 "决策清单"展示结果）+ "求解器监控"管理页

**界面元素与展示规则**（管理页）:

| 元素 | 类型 | 默认态 | 操作后 | 禁用条件 |
|---|---|---|---|---|
| 当前任务进度 | ProgressBar | 0%（无任务）/ 实时百分比 | hover 显示当前阶段（loading/preprocessing/solving/postprocess） | - |
| 历史任务表 | Table | 最近 30 天，倒序 | 列：runId/触发方式/SKU 数/求解时长/状态/decision_id | - |
| 求解器配置 | Panel | 显示当前 backend（SCIP/Gurobi）+ time_limit + gap_tolerance | 改后下次任务生效 | 非 admin 只读 |
| 失败任务详情 | Modal | - | 显示日志 + 求解器 stderr + 重跑按钮 | - |
| 求解性能图表 | Chart | 30 天求解时长趋势 | hover 显示当日详情 | - |

**交互逻辑**:
1. **求解器选型**（响应工程 R3 N1）：v1 默认开源 SCIP（Python via `pyscipopt`）；如 license 允许 fallback Gurobi（更快）；接口抽象为 `IOptimizer`。
2. **问题形式化**：
   - 决策变量：`x_s,t,m ∈ Z_+`（SKU s × 时间窗 t × 物流方式 m）；时间窗 t 用周聚合（13 周 = 90 天）降维。
   - 目标：max Σ (revenue - cost components)
   - 约束：库存平衡递推；FBA Restock Limit；总资金；批量经济量（订单最小起订）。
3. **维度控制**：200 SKU × 13 周 × 2 物流 = 5200 整数变量（远低于 36000 周日级，对应 §10.B 性能需求）。
4. **预处理**：对超低需求 SKU（FR-16 长尾）跳过优化直接走规则；对当前库存 >120 天可卖的 SKU 跳过。
5. **后处理**：将 t（周）拆分回当周内的最早可发日；输出含 per-SKU 的成本/收益拆解（喂给 FR-6 可解释面板）。
6. **接口阻抗**（响应工程 R3 N2）：v1 用 p50 求基础解；用 p10/p90 各跑一次得到"乐观/悲观"对照解；输出"基础决策 + 三场景差异"。Robust optimization 留 v1.5。
7. **缓存**：相同 input_hash（snapshot_id + params_version + 预测版本）24h 内命中缓存。

**异常场景**:

| 类型 | 场景 | 处理 |
|---|---|---|
| 数据异常 | 模型 infeasible（约束冲突，如 Restock Limit < 必须补量） | 自动放松"软约束"（按优先级：先放总资金 +20%，再放 Restock Limit），并在结果里标注"约束放松" |
| 数据异常 | 求解器返回 unbounded | 阻断；通知管理员；fallback 用上次决策 |
| 第三方失败 | 求解器进程崩溃 / OOM | 监控 + 自动重启 1 次；二次失败上报 P0 |
| 并发冲突 | 同一 snapshot 被两个调度同时触发 | 分布式锁（Redis SETNX, TTL 30min）；后到的等待并复用结果 |
| 性能 | 求解超时（>30 min） | 接受次优解（current best），并标"未达最优 gap = X%" |
| 用户操作冲突 | 用户点"重跑"时上一个任务未结束 | 提示"上一个任务进行中"或选"取消并重跑" |
| 设备差异 | 管理页移动端宽度不足 | 关键列固定，详情进二级页 |

**边界数值**:
- 单次求解时长上限：1800 s（30 min）
- gap 容忍：默认 1%（即接受 1% 内的次优）
- 内存峰值：< 8 GB
- 并发任务上限：1（v1 单租户单实例）
- 求解输入缓存 TTL：24 h

### 5.6 FR-5 灵敏度分析模块

**交互逻辑**:
1. **执行模式**：each-at-a-time（一次扰动一个参数）+ 关键参数组合扫描（默认 3 个关键参数的 3³ = 27 个场景）。
2. **核心决策稳定率定义**："核心决策" = top-30 总成本 SKU 的"是否发货 + 物流方式" 二元决策；稳定率 = (扰动后与基础一致的核心决策数) / 30。
3. **结果存储**：灵敏度 run 是独立 entity（`sensitivity_runs`），关联到基础 decision_id；保留 90 天。
4. **触发时机**：每日基础决策后自动跑一次（凌晨 03:00 UTC）；用户也可手工触发。
5. **可解释面板**联动：FR-6 单条建议里嵌"该建议在 ±20% 扰动下稳定/不稳定"小图标。

**异常场景**:

| 类型 | 场景 | 处理 |
|---|---|---|
| 性能 | 27 场景 × 30 min = 13.5h 超长 | 默认采用启发式：基础解为热启动 + warm-start MIP；目标 27 场景 < 60min |
| 数据异常 | 某个扰动场景 infeasible | 该场景标"infeasible"，不计入稳定率分母 |
| 并发冲突 | 用户手工触发与定时任务冲突 | 队列化 |
| 第三方失败 | 求解器在第 N 个场景失败 | 跳过该场景，记 log；不影响其他 |
| 设备差异 | 龙卷风图在小屏失真 | 切换为表格视图 |

**边界数值**:
- 单场景求解时长（heuristic）：≤ 120 s
- 总 batch 时长上限：3600 s（1h）
- 稳定率统计 SKU 数：top 30（按总成本）

### 5.9 FR-8 缺货预警系统

4. **推送渠道**（响应商业 R1 B6）：默认企微应用 / Bark / Telegram；每渠道独立模板；失败 fallback 到下一个渠道。
5. **静音**：写 `mute_rules`（scope=sku|global, until=ts）；推送前查 rule。
6. **SLA**：从触发到推送 ≤ 15 分钟（覆盖锚点 §3.2 的"立刻"，与原"1 小时"冲突取严）。

**异常场景**:

| 类型 | 场景 | 处理 |

| 第三方失败 | 企微 token 过期 | 自动尝试 refresh；失败标渠道为"异常"并通知 admin |
| 并发冲突 | 同时触发与静音设置 | 静音以触发时刻 query 为准 |
| 权限不足 | 助理改聚合配置 | 403 |
| 设备差异 | 移动端弹窗推送 | iOS 走 Bark，Android 走企微 |
| 用户操作冲突 | 静音规则与紧急 critical | critical 级别可配置"穿透静音"（默认开） |

**边界数值**:
- 触发到推送 SLA：≤ 15 min（含聚合等待）
- 单 SKU 同类预警节流：1 小时内最多 1 条
- 历史保留：180 天
- 推送渠道并发：3 个并行

### 7.2 并发与冲突

| 场景 | 策略 |
|------|------|
| 同决策被多人拍板 | 乐观锁 + version 字段；后到 409 + 显示差异 |
| 同 SKU 多预警短时触发 | 5min 窗口聚合 |
| 同数据源被多个 sync 触发 | 分布式锁（Redis）+ 去重 |

| 跨 tab 操作 | 通过 WebSocket 广播 + 本地 toast 通知"另一端已修改" |

### 7.3 第三方依赖失败

| 依赖 | 失败模式 | 缓解 | 监控 |
|------|---------|------|------|
| FBA Reports API | 5xx / 限流 / 超时 | 指数退避 + 上次快照 fallback + P1 告警 | 成功率 / 平均延迟 |
| 求解器（SCIP/Gurobi） | OOM / 崩溃 / license 过期 | 自动重启 + 双引擎 fallback + license 7 天前预警 | 求解时长 / 失败率 |

| 货代/海运/空运报价 | v1 无真实接入 | 用户手工录入 + 周更提醒 | 报价新鲜度（>7 天 warning） |

| WebSocket（实时推送） | 断连 | 自动重连 + 30s 轮询兜底 | 连接稳定时长 |
| 数据库 | 主库故障 | 读从库降级（只读模式 UI） + P0 告警 | 主从延迟 |

## §9.F 上线后稳定性风险

| 风险 | 缓解 | 监控 |
|------|------|------|
| 求解器内存泄漏 → OOM | 进程级隔离 + 自动重启 + max_memory_limit | `process.memory_rss` 持续 > 6GB 报警 |
| 推送渠道大面积失败 | 多渠道 fallback + 失败队列 + 站内信兜底 | 各渠道成功率 < 95% 报警 |
| 决策连续多日不准 | 7 天滚动正确率 < 50% 自动降级朴素策略 | `decision.weekly_accuracy` |
| 助理状态机卡死（如 in_progress 7 天不变） | 自动告警 + CEO 视图标"长时未更新" | 状态停留时长 monitoring |
| 数据源 schema 突变（FBA API 升级） | Schema 校验 + 失败硬阻断 + 通知 admin | schema_version_mismatch_count |
| 时区切换日异常（夏令时） | 单独测试套件 + 切换日前 24h 预演 | `business_date_consistency_check` |
| 影子模式被错当 Mock 用 | UI 顶部独立紫色标识 + 决策不进 FR-7 | `mode_misuse_attempts` |
| 用户连续 N 天不打开 | 周报邮件兜底 + 紧急 critical 必推 | `user.last_active` |
| 数据回流延迟导致复盘卡住 | 复盘"待回流"标识 + 14 天硬切 | `playback.pending_count` |
| 并发改参数导致目标函数发散 | 参数版本快照 + self-test 50 case 阻断发布 | self-test failure 报警 |
| 长尾 SKU 被错分类 → 不出建议 | FR-16 人工覆盖 + 周复盘必看 | `sku.no_recommendation_streak` > 30 天 |
| 批量接受失误（高比例错误） | 周复盘批量正确率 < 80% 自动提示调整条件 | `batch.accuracy` |
| MINLP 在某 SKU 组合上 infeasible | 软约束自动放松 + 标"约束放松" | `solver.infeasible_count` |
| 助理离职数据丢失 | 账号停用前导出待办；自动归还 CEO 池 | account_disable_audit |
| 法律/合规：商业数据走个人微信 | 默认渠道企微 / Bark / Telegram，禁用个人微信 | 渠道配置审计 |

### 12.4 关键架构决策记录（ADR 索引，简版）

| ADR | 决策 | 替代方案 | 选择原因 |
|-----|------|---------|---------|
| ADR-01 | 求解器用 SCIP（开源） | Gurobi / CPLEX | 0 license 成本；v1 规模可承载 |
| ADR-02 | 区间预测用分位数回归 LightGBM | Prophet / DeepAR | 工程成熟 + 单 SKU < 30s |
| ADR-03 | 预警与日报不合并 | 合并收件箱 | 状态机/SLA 完全不同（响应工程 R3） |
| ADR-04 | Mock 与 Shadow 物理隔离 | 共用 DB + tenant_id | 资金池/隔离更强；避免数据污染 |
| ADR-05 | UTC 入库 + PST 业务日 | 单一时区 | 多站点扩展性 |
| ADR-06 | 周决策 + 事件触发预警 | 每日决策 | R3 共识 "200 SKU 健康业务实际每周 2-3 次发货" |
| ADR-07 | 求解输入用 input_hash 缓存 | 不缓存 | 24h 复跑可立即命中，节省 90% 算力 |
| ADR-08 | 参数版本快照绑定决策 | 当前最新值 | 复盘可复现 |
