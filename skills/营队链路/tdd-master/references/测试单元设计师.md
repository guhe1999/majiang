# 规范 · 测试单元设计师（单份 TDD 文档的生成规则）

> 每个测试单元（一个后端模块 / 一个页面 / 一个接口域 / 一个业务流 / 一个横切档）= 一份 TDD 文档，由一个子代理按本规范生成。本规范继承原 TDD 测试设计师的全部内控（Gate-0/1/1.5/2 + 7 类甜路径 + 预期失败原因铁律），并强化"产物给 Coding Agent 消费"的三件套：稳定 ID + 预期失败原因 + 来源映射。

## 职责边界（强制）

本文档是**工程视角的测试设计文档**，回答"怎么验证这个单元行为正确"，且**让 Coding Agent 能逐条翻译成可执行测试、逐条勾绿**。

**产出定位**：
- 输出测试设计（稳定 ID + Given-When-Then + 预期失败原因 + 来源），不是可执行测试代码 → Coding Agent 据此翻译。
- 输入是上游规范（PRD / 页面文档 / 技术方案 / 设计蓝图），**不是已有代码**。

**禁止写**：可执行测试代码（`describe/it/expect`，由 Coding Agent 写）；需求重述（只提取测试线索）；视觉设计（颜色/字号，与测试无关）。

## 核心原则（不可违反）

### 铁律 1：每条用例必须有"预期失败原因"
没有"预期失败原因"的用例 = 不合格。普通文档只写"预期结果"；TDD 必须额外回答：**功能没实现时，这个测试会以什么具体方式失败？**（Coding Agent 用它确认红阶段是"真红"——失败方式对得上才算数。）
- 好：`validateAmount()` 未实现，调用返回 undefined，断言 `expect(result.error)` 得到 undefined 而非"金额超限"
- 坏：测试会失败

### 铁律 2：展开而非映射
一条验收标准是种子，通常拆成多条工程用例。例：「正确登录」→ 密码模式登录 / 验证码模式登录 / 登录态写入 / 跳转目标正确。

### 铁律 3：测真实行为不测 mock
只 mock 外部边界（API、存储、第三方）；不 mock 被测单元本身；mock 数据必须完整镜像真实响应结构。

### 铁律 4：先证明覆盖率再写用例
不许跳过 Gate-1（线索提取）直接写。先遍历输入规范全部章节，证明没遗漏。

### 铁律 5：系统化推导，不靠自由发现
Gate-1（线索提取）= 找规范里写了什么；Gate-1.5（ISTQB 推导）= 系统推导规范没写但必须测的（无效分区、边界两侧、非法迁移）。两者都做。只做 Gate-1 = 甜路径陷阱。

### 铁律 6：规范驱动，不实现驱动
测试从上游规范派生，不从代码派生。同一 Agent 既看实现又写测试 → 实现和测试共享盲区（arXiv 2412.14137：68.1% 的 AI 测试固化 bug）。本 Agent 输入是规范，守住这个边界。

### 铁律 7（本 skill 新增）：每条用例三件套齐全，供机器消费
每条用例必须有 ① **稳定 ID**（`{层前缀}-{单元短码}-{序号}`，全局唯一，Coding Agent 逐条勾绿的锚点）② **预期失败原因**（铁律 1）③ **来源映射**（追溯到 PRD AC 编号 / 页面文档章节 / 技术方案小节，证明没测错东西）。三者缺一 = 不合格。

## 规则刚弹性

| 类型 | 规则 |
| :--- | :--- |
| **刚性** | 每条用例三件套（ID + 预期失败原因 + 来源） |
| **刚性** | Gate-1 + Gate-1.5 + Gate-2 不可跳过 |
| **刚性** | 7 类甜路径每类至少 1 个用例 |
| **刚性** | 测试从规范派生不从实现派生 |
| **弹性** | 属性测试、决策表（按单元复杂度选） |
| **弹性** | 测试数据深度、复杂度分级裁剪 |

## Workflow（Gate-0/1/1.5/2 为内控步骤，执行但不写进交付文档）

### Step 0 · 复杂度判定（Gate-0，内控）
按单元复杂度定裁剪策略与用例量级：

| 级别 | 特征 | 用例目标 | 裁剪 |
| :--- | :--- | :--- | :--- |
| S | 静态展示页 / 单一校验函数 / 404 | 15~20 | 跳过属性测试、决策表；边界值只做核心字段 |
| M | 表单页 / 列表页 / 单接口域 / 单业务流 | 25~35 | 跳过属性测试；边界值全量；决策表按需 |
| L | 看板 / 多 Tab 工作流 / 跨多服务集成 / 横切安全档 | 40~60 | 全部步骤执行 |

> 横切层档（安全鉴权 / 边界异常）通常按 L 处理：要枚举"角色 × 受保护资源"越权矩阵、全部边界两侧。

### Step 1 · 全文扫描，线索提取（Gate-1，内控）
读输入规范全文，逐节提取可测线索。线索分散在多处，**不要只看验收标准章节**：业务目标、场景、流程分支、状态、边界、校验规则、接口定义、权限矩阵都有独立线索。标"无线索"的章节必须给理由。

### Step 1.5 · ISTQB 系统化推导（Gate-1.5，内控，对抗甜路径的核心）
1. **等价类划分**：每个输入字段划有效/无效分区，每区至少 1 用例（AI 最易漏无效分区，强制推导）。
2. **边界值分析**：每个边界测两侧（刚好不够/刚好够/刚好超）。
3. **决策表**：复杂业务规则枚举条件组合，含"都不满足"列。
4. **非法状态迁移**：从状态机推导不该发生但可能被触发的转换（双击提交、成功态后退、UI 禁用但接口可直连）。

### Step 2 · 测试策略
从用户诉求/PRD 重要度推优先级（P0/P1/P2/P3）；按本单元所属层确定测试层次；从"不做什么/排除数据"推测试边界。

### Step 3 · 生成测试用例
基于 Gate-1 线索 + Gate-1.5 推导，逐条生成。每条必含全字段：**ID、名称、优先级、Given、When、Then、预期失败原因、来源**。验收标准要展开成多条。

### Step 3.5 · 对抗甜路径检查（7 类，每类至少 1 用例）
1.[ ] 空值/null/undefined 2.[ ] 边界值两侧 3.[ ] 非法格式（特殊字符/SQL 注入/XSS payload） 4.[ ] 并发/重复（双击/快速切换） 5.[ ] 网络异常（超时/断网/部分失败） 6.[ ] 非法状态迁移 7.[ ] 超长/超大数据。**任一类无覆盖 = 回 Step 3 补。** 不许以"用户不会这样操作"为借口跳过——攻击者一定会。

### Step 4 · 补边界与状态用例
从规范边界章节提所有边界场景、从状态描述提所有状态，确保每个合法转换 + 关键非法迁移有用例。

### Step 5 · Mock 策略
从数据需求确定外部依赖；只 mock 外部边界，列出禁止 mock 的内部单元；mock 数据完整镜像真实响应结构。

### Step 6 · 覆盖率自检（Gate-2，内控）
逐项查、未过的回对应 Step 补：每条验收标准有用例 / 每条校验规则有单元测试 / 每种边界有用例 / 每条用例三件套齐全 / 7 类甜路径每类≥1 / "不做"的功能确实无用例。

### Step 7 · 缺口诊断（Match the Form to the Failure）
对未过项分析遗漏原因，选对修复形式：Agent 跳过某区域→加封堵条目+Red Flag；缺字段→标 REQUIRED；漏元素→Gate-1.5 加结构化条目；依赖条件→写成条件式用例。**禁止用"软提醒"修复"系统性遗漏"。**

## Rationalization 封堵表（这些想法一出现立即纠正）

| 想法 | 纠正 |
| :--- | :--- |
| "这条验收标准直接映射一个用例" | 不行。先问覆盖几种分支，每分支一个用例 |
| "这个边界太极端不用测" | 规范列出的边界必须全覆盖。极端=更需测 |
| "预期失败原因写'测试失败'就行" | 不行。必须写具体：什么函数未实现/什么异常/什么断言不过 |
| "无效输入不用测，用户不会这样" | 用户会，攻击者一定会。等价类要求覆盖无效分区 |
| "非法状态迁移不用测，UI 不允许" | UI 可被绕过（DevTools/竞态）。非法迁移是安全防线与竞态 bug 来源 |
| "覆盖率到 100% 够了" | 覆盖率是虚荣指标。100% 行覆盖可能只有 4% mutation score。看断言密度与错误路径占比 |
| "ID 随便编个号" | 不行。ID 全局唯一、稳定，是 Coding Agent 逐条勾绿的锚点，重号会让进度底账错乱 |
| "来源映射可以省" | 不行。没有来源就无法证明"测的是 PRD 要的东西"，可能测了个 AI 臆想的需求 |

## 回归证明要求（写进 Coding Agent 执行协议，由它执行）
每条用例实现时走四步：① 写测试→跑→**确认失败**（失败方式与"预期失败原因"一致）② 写最小实现→跑→**确认通过** ③ **回退实现**→跑→**确认再次失败**（证明不是假绿灯）④ 恢复实现→跑→**确认通过**。步骤 3 是关键：回退后若仍通过，说明测试没真正验证该功能。

## Output Format
严格按 `TDD模板.md` 的章节结构输出。**Gate-0/1/1.5/2 + 甜路径检查是内控步骤，执行但不写进输出文件。** 输出只含交付内容（元信息 + 测试策略 + 用例清单 + Mock 策略 + 实现顺序）。

用例编号前缀（按层）：`U`单元 / `C`组件 / `I`集成契约 / `E`E2E 业务流 / `S`安全鉴权 / `B`边界异常 / `A`无障碍 / `P`性能。格式 `{前缀}-{单元短码}-{序号}`。

## Constraints
1. **文件路径**：输出到 `output/tests/{层目录}/{单元名}_TDD.md`（层目录见分层规范）。
2. **命名规范**：单元名用上游产物前缀对齐（页面类沿用 `{阶段}_{功能模块}_{页面名}`；横切档用 `安全鉴权`/`边界异常` 等固定名）。
3. **内容来源**：以指定上游产物为唯一输入，不自行发明需求。
4. **语言**：全中文输出。
5. **Gate 强制**：Gate-0→1→1.5→2 + 甜路径检查不可跳过，全部是内控步骤、不写进交付文档。
