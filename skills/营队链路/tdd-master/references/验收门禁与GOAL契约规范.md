# 规范 · 验收门禁与 GOAL 契约（给 Coding Agent 消费）

> 这是 tdd-master 与其它 master 的根本不同之处的落点：产物不是给人看的设计稿，而是给 Claude Code / Codex 等 Coding Agent 在 `/go` GOAL 模式下消费的**验收契约**——它定义"什么叫做完了"，是开发的终止条件。本规范定义两份机器可解析产物的格式：`测试索引.md`（全量用例底账）+ `验收门禁与GOAL契约.md`（总入口）。

## 设计目标：让 Coding Agent 能"无人确认地跑到全绿"

Coding Agent 的循环是：读门禁契约 → 知道完成定义 → 逐条读用例 → 红（确认失败方式对得上）→ 绿（最小实现）→ 重构 → 在索引里勾绿 → 直到完成定义满足。**全程不需要人再做决策**——因为所有决策已在 tdd-master 阶段 I 跟用户敲死并固化进契约。所以两份产物必须：① 稳定可解析（固定表头、全局唯一 ID）② 自带完成定义（分级闸）③ 自带消费协议（怎么跑、怎么算交付）。

---

## 产物 A · `测试索引.md`（全量用例底账）

主代理扫所有 `tests/**/*_TDD.md`，抽每条用例汇成一张总表。这是 Coding Agent 逐条勾绿的进度账，也是校验"有没有重号/漏层"的依据。

````markdown
# 测试索引（全量用例底账）

> Coding Agent：这是所有待转绿用例的总清单。按优先级实现，每转绿一条在"状态"列标 ✅。
> 全部 P0 + P1 转绿 = 满足完成定义（见 验收门禁与GOAL契约.md）。

## 汇总
- 选中层：单元 / 组件 / 集成契约 / E2E 业务流 / 安全鉴权 / 边界异常（共 6 层）
- 用例总数：NN（P0: a / P1: b / P2: c / P3: d）
- 交付闸：全部 P0(a 条) + P1(b 条) 转绿

## 用例清单
| ID | 层 | 优先级 | 用例名称 | 来源 | 所在文件 | 状态 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| S-AUTH-01 | 安全鉴权 | P0 | 普通用户越权访问他人订单被拒 | 设计蓝图 §角色权限 | crosscut/安全鉴权_TDD.md | ☐ |
| E-ORDER-01 | E2E | P0 | 登录→下单→支付→看到订单号 全链路 | PRD §业务场景 | e2e/下单_业务流TDD.md | ☐ |
| U-PRICE-02 | 单元 | P1 | 金额超上限返回错误码 | 技术方案 §数据模型 | unit/price_TDD.md | ☐ |
| …  |  |  |  |  |  | ☐ |
````

**生成规则**：
- 表头固定不变（机器按列解析）。
- ID 全局唯一；发现重号回对应子代理重排。
- "状态"列初始全 ☐，Coding Agent 实现后改 ✅（或单独维护进度文件，不污染本表亦可）。

## 产物 B · `验收门禁与GOAL契约.md`（Coding Agent 总入口）

Coding Agent **第一份要读的文件**。结构固定如下：

````markdown
# 验收门禁与 GOAL 契约 · {{产品名}}

> 给 Coding Agent：这是本项目的开发终止条件。按本契约用 TDD 把所有用例转绿，
> 满足"完成定义"那一刻才算交付。所有测试设计已在 tests/ 下分层给出，无需再问人。

## 1. 完成定义（= /go GOAL 终止条件）⭐
- **交付闸**：`测试索引.md` 中全部 **P0 + P1** 用例转绿（共 X 条）。P2 尽力、P3 可延后。
- **安全层特例**：安全鉴权层全部 P0 **零失败、不可降级**——有一条不过即未交付。
- **覆盖率门槛**：业务核心模块 ≥90% / 基础设施 ≥80% / UI 胶水 ≥60%。
- **变异测试（可选）**：mutation score ≥70%（防假绿灯；蓝图勾选才启用）。
- 以上任一不满足 = 未交付，继续循环。

## 2. 选中层与文件
| 层 | 用例数(P0/P1/P2) | 文件 |
| :--- | :--- | :--- |
| 单元 | .. | tests/unit/*_TDD.md |
| 组件 | .. | tests/component/*_TDD.md |
| 集成契约 | .. | tests/integration/*_TDD.md |
| E2E 业务流 | .. | tests/e2e/*_TDD.md |
| 安全鉴权 | .. | tests/crosscut/安全鉴权_TDD.md |
| 边界异常 | .. | tests/crosscut/边界异常_TDD.md |

## 3. 上游产物（实现时一并读，但测试只从规范派生）
- 业务规则与验收：output/PRD详细版.md
- 技术栈/架构/API/安全：output/技术方案.md
- 角色权限/站点联动：output/设计决策蓝图.md
- 页面行为/状态/边界：output/pages/*.md（+ *.html 高保真参考）

## 4. Coding Agent 消费协议（怎么跑）
1. 先读本契约 §1 完成定义，再读 测试索引.md 拿到全量用例。
2. 按 **P0 → P1** 顺序，逐条用例：
   - 读用例的 Given-When-Then + **预期失败原因**。
   - 写测试 → 跑 → **确认失败，且失败方式与"预期失败原因"一致**（不一致说明测试写错或理解错，停下修正，别急着写实现）。
   - 写最小实现 → 跑 → 确认转绿。
   - **回退实现 → 跑 → 确认再次失败**（证明不是假绿灯）→ 恢复 → 确认绿。
   - 在 测试索引.md 把该 ID 标 ✅。
3. 每完成一层，跑该层全量回归 + 覆盖率；达 §1 门槛再进下一层。
4. 全部 P0+P1 ✅ 且覆盖率/变异达标 → **交付**。否则回第 2 步。
5. **测试只从 tests/ 的规范派生，不从已写的实现反推**——避免实现与测试共享盲区。

## 5. 测试金字塔比例参考
- 单元/组件 ~70% · 集成契约 ~20% · E2E ~10%；横切层（安全/边界/无障碍/性能）贯穿，按 §1 闸独立卡。
````

**生成规则**：
- §1 完成定义直接取《测试决策蓝图》§3，逐字落地（这是用户阶段 I 拍板的，不可改）。
- §2 / 测试索引汇总数由主代理扫 `tests/**/*_TDD.md` 实际统计，不估算。
- §4 消费协议是固定模板，原样写入。

## 与 `/go` GOAL 模式的衔接（写给用户的一句话）

> 交付后告诉用户：把 `{项目名}/output/` 整个目录交给 Claude Code，指令——
> "读 tests/验收门禁与GOAL契约.md，用 /go GOAL 模式按其消费协议把所有 P0+P1 用例转绿，全绿前不要停。"
> GOAL = 契约 §1 完成定义；Coding Agent 自己循环红-绿-重构直到 GOAL 满足，那一刻才算真正交付。

## 自校验（主代理产出后必查）
- [ ] 测试索引.md 用例数 = 各 TDD 文档用例数之和（无遗漏）。
- [ ] ID 全局唯一无重号。
- [ ] 每条用例三件套齐全（ID + 预期失败原因 + 来源）。
- [ ] 契约 §1 完成定义与《测试决策蓝图》§3 逐字一致。
- [ ] 契约 §2 各层文件路径真实存在（Bash 核查）。
