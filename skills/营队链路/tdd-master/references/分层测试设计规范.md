# 规范 · 分层测试设计（测试金字塔 8 层）

> 本规范定义 tdd-master 的测试分层全集。阶段 I 元选择用它把"测哪几层"摊给用户；阶段 II 用它把每层拆成测试单元并定位输入产物。**目标：从功能、安全、页面联动、边界四个维度，全栈全覆盖。**

## 为什么分层（而不是"每页一份 TDD"）

旧做法以"页面"为单位 → 测试天然偏前端、只测"页面报错级"行为；后端逻辑、API 契约、跨页业务流、登录鉴权、越权这些**横切系统正确性**的东西没有载体。分层把测试按**测试金字塔**重组：底层多而快（单元/组件）、中层验证集成与契约、顶层验证业务流，横切层（安全鉴权/边界/无障碍/性能）贯穿所有层独立成档。

```
        ╱╲   E2E 业务流(E)      ← 跨页端到端：登录→鉴权→核心交易→反馈
       ╱──╲       ~10%
      ╱────╲  集成+契约(I)      ← API↔DB 真实副本、前后端接口 schema
     ╱──────╲      ~20%
    ╱────────╲ 单元(U)+组件(C)  ← 后端领域逻辑 + 前端纯函数/组件
   ╱──────────╲     ~70%
  ══════════════════════════════
   横切层(贯穿所有层，各自独立成档)：
   安全鉴权(S) · 边界异常(B) · 无障碍(A) · 性能基线(P)
```

## 8 层定义表

| 层 | 编号前缀 | 测什么 | 输入来源（上游产物） | 拆分单位（= 一份 TDD 文档） | 默认 |
| :--- | :---: | :--- | :--- | :--- | :---: |
| **L1 单元** | `U` | 纯逻辑：后端领域规则/算法/校验函数、前端纯函数/数据转换/状态计算。不碰 IO | 技术方案.md（数据模型/算法/校验）+ PRD §算法 + pages §校验规则 | 按后端模块 / 前端逻辑模块 | 核心 |
| **L2 组件** | `C` | 单个前端组件的渲染、交互、状态切换（空/载/错/无权限） | pages/*.md（§区域/§状态/§交互） | 按 MVP 页面 | 核心 |
| **L3 集成+契约** | `I` | API↔DB（真实依赖副本：test DB / mock server）、服务间调用；**前后端接口契约**：请求/响应 schema、状态码、错误码 | 技术方案.md（§架构/§API 接口）+ pages §数据需求 | 按接口域（一组相关 API） | 核心 |
| **L4 E2E 业务流** | `E` | 跨页面端到端业务链：完整用户旅程（如 登录→进工作台→建内容→提交审核→通过）。验证**页面联动** | 页面清单.md（调用关系）+ 设计蓝图（站点地图）+ PRD §业务场景 | 按核心业务流（用户旅程） | 核心 |
| **L5 安全鉴权** | `S` | 登录/登出/会话/token 生命周期、RBAC 角色边界、**水平越权**（看他人数据）、**垂直越权**（伪造高权角色）、注入(SQL/NoSQL)、XSS、CSRF、敏感数据脱敏、关键参数前端篡改后端是否校验（如支付金额） | 技术方案.md（§认证/§安全）+ 设计蓝图（§角色权限）+ OWASP Top 10 | 横切，**1 份独立档**（`crosscut/安全鉴权_TDD.md`） | 核心 |
| **L6 边界异常** | `B` | 空值/null/undefined、边界值两侧（刚好够/刚好不够/刚好超）、并发与重复操作（双击提交、快速切换）、网络异常（超时/断网/部分失败）、超长超大数据、非法状态迁移（UI 不允许但可被绕过） | pages §边界 + Gate-1.5 ISTQB 系统推导 + 技术方案 §约束 | 横切，**1 份独立档**（`crosscut/边界异常_TDD.md`） | 核心 |
| **L7 无障碍** | `A` | 键盘可达、焦点管理、屏幕阅读器/ARIA、对比度、表单标签关联 | pages §无障碍验收 | 横切，**1 份独立档** | 可选 |
| **L8 性能基线** | `P` | 关键接口 SLA、首屏/响应时间、并发承载、大列表渲染 | 技术方案 §性能 + PRD §非功能需求 | 横切，**1 份独立档** | 可选 |

> 阶段 I 元选择预设档：**全 8 层（推荐）** / **核心 5 层 = L1+L2+L3+L4+L5** / **精简 3 层 = L1+L4+L5**。

## 各层与"四维度"的对应（回答用户"功能/安全/联动/边界都要测"）

| 用户要的维度 | 由哪些层承接 |
| :--- | :--- |
| **功能正确** | L1 单元 + L2 组件 + L3 集成契约 |
| **安全 / 登录** | L5 安全鉴权（登录、会话、越权、注入全在这层） |
| **页面联动** | L4 E2E 业务流（跨页端到端链路） + L3 契约（前后端不脱节） |
| **边界管理** | L6 边界异常（系统化 ISTQB 推导，专治 AI 甜路径盲区） |

## 横切层为什么必须独立成档（不塞进页面）

登录、鉴权、越权、注入、边界——这些不属于任何单一页面，**散到各页面就会被切碎、漏测**。独立成档才能：① 系统化覆盖（一次性枚举所有角色×所有受保护资源的越权矩阵）；② 在阶段 I 一次性跟用户问清"跨哪些页面/接口生效"；③ Coding Agent 把它当一组独立验收闸统一跑。

## 编号规则（全局唯一，供测试索引与 Coding Agent 逐条勾绿）

- 前缀按层：`U`/`C`/`I`/`E`/`S`/`B`/`A`/`P`。
- 格式：`{前缀}-{单元短码}-{两位序号}`，如 `S-AUTH-01`（安全鉴权第 1 条）、`E-ORDER-03`（下单业务流第 3 条）、`U-PRICE-02`。
- **全局不重号**：主代理汇总 `测试索引.md` 时校验唯一性，重号回对应子代理改。
- 优先级 `P0/P1/P2/P3` 从 PRD §验收标准重要度 + 用户阶段 I 门禁确认推导（安全层 P0 默认不可降级）。
