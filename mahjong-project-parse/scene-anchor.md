# 场景锚点（最终版）

## 产品名（暂定）

Mahjong Project Parse

## 一句话定位

把 `code-back/mahjong` 从“已有但尚未系统理解的 Go 麻将后端代码”整理成“可学习、可协作、可继续开发、可验收”的项目知识包。

## 核心用户

- lei：负责搭建项目、提取设计方案、学习源代码，需要清楚项目结构、模块职责、运行/依赖方式和后续开发路径。
- he：负责提出新需求并推动项目更新，需要知道现有系统能力边界、需求影响范围和验收标准。

## 核心场景

团队要继续维护或扩展 `code-back/mahjong`，但在直接改业务代码前，需要先建立可信的项目地图、文档体系、迁移计划和验收契约。

## 已确认范围

第一版只做：

- 项目解析。
- 文档沉淀。
- 迁移/整理计划。
- TDD 与验收契约。

第一版不做：

- 不直接修改 `code-back/mahjong` 业务代码。
- 不重构麻将核心逻辑。
- 不直接开发前端业务功能。

## 成功定义

第一版完成后：

- lei 能根据文档讲清 `code-back/mahjong` 的源码结构、模块职责、核心调用关系和后续搭建路径。
- he 能根据文档提出新需求，并初步判断需求会影响哪些模块、流程和验收点。
- 团队能基于验收契约判断“解析/文档/计划”是否完整，而不是凭感觉认为完成。

## 解析深度

第一版需要深入到核心流程，至少覆盖：

- 项目入口与包结构。
- 模块职责和依赖关系。
- 房间生命周期。
- 用户状态与行为。
- 牌局运行流程。
- RPC 或外部调用边界。
- 匹配/玩法相关逻辑。
- 异常、容错、恢复或任务调度相关逻辑。

## 目录映射

- 链路产物：`mahjong-project-parse/output/`
- 最终项目文档：同步或整理到 `document/`
- 对话与关键决策：`command/`
- 脚本：`script/`
- 后端原始项目：`code-back/mahjong`
- 后续前端代码：`code-front/`

## 待确认

无。

## 验收口径

第一版必须同时满足：

- 文档完整性：核心文档齐全，覆盖入口、模块、核心流程、外部边界、迁移计划和验收契约。
- 源码引用：关键判断能回指到具体源码文件或目录。
- 影响范围表：后续新需求能通过影响范围表初步判断会改哪些模块和流程。
- 验收契约：有明确的完成定义、检查清单和后续 Coding Agent 可消费的门禁入口。

## 脚本边界

第一版提供轻量扫描/索引脚本，用于辅助生成或校验文件清单、模块索引、文档同步状态。不做复杂自动化文档生成，不做代码迁移自动化，不修改业务代码。
