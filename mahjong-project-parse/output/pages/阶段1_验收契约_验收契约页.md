## 2. 页面概述

### 2.1 一句话定位

验收契约页定义第一版解析、文档、影响范围和脚本交付是否完成。

### 2.2 存在理由

1. 没有契约，完成与否只能凭感觉。
2. 没有源码引用检查，文档可能不可信。
3. 没有 TDD 入口，后续链路无法稳定承接。

### 2.3 页面目标（可衡量）

| 目标维度 | 具体指标 |
| :--- | :--- |
| 门禁完整 | 覆盖文档、源码引用、影响范围和脚本 |
| 可执行 | 每条 AC 有优先级和测试数据 |
| 可承接 | 能指向 TDD master 下一步 |

### 2.4 不做什么

- 不执行真实测试。
- 不自动修改文档。
- 不判定业务逻辑正确性。

## 3. 用户与场景

### 3.1 目标用户

| 维度 | 描述 |
| :--- | :--- |
| 角色 | lei / he / Coding Agent |
| 技术熟练度 | 能理解验收清单 |
| 使用设备 | 桌面浏览器 |
| 使用姿势 | 交付前逐项检查 |

### 3.2 核心使用场景

**场景 A：确认第一版是否完成**
- 时机：设计和文档产物生成后。
- 前置：核心文档和页面原型已存在。
- 动作：查看 AC 清单 → 处理未通过项。
- 情绪：希望有明确门槛。

**场景 B：进入 TDD master**
- 时机：设计阶段交付后。
- 前置：页面文档和技术方案已生成。
- 动作：确认门禁 → 进入测试设计。
- 情绪：希望交接稳定。

### 3.3 用户核心诉求

1. 知道完成定义。
2. 找到未通过项。
3. 稳定交给 TDD。

## 4. 信息架构

### 4.1 区域划分

1. 完成定义。
2. 门禁状态。
3. AC 清单。
4. 未验证结论。
5. 下一步入口。

### 4.2 区域内容

**AC 清单**
- 文档完整性。
- 源码引用。
- 影响范围表。
- 脚本边界。
- TDD 入口。

## 5. 界面内容

### 5.1 元素清单

**完成定义**
- 内容：完成 F1-F7 即 MVP 完成。
- 数据来源：PRD。

**门禁状态**
- 状态：通过、待补充、高风险。
- 数据来源：验收契约。

**下一步入口**
- 文案：“进入 TDD master”。
- 显示条件：门禁通过或人工确认可继续。

### 5.2 辅助信息

- 未验证结论清单。
- 最后检查时间。
- Git 状态提示。

## 6. 交互流程

### 6.1 页面加载与初始化

- 读取 AC 清单。
- 统计通过和未通过数量。

### 6.2 核心操作流程

```text
打开验收契约 → 查看门禁状态 → 处理未通过项 → 确认可进入 TDD
```

### 6.3 辅助操作

- 按优先级筛选。
- 复制 AC。
- 跳转缺失文档页面。

## 7. 状态与边界

### 7.1 页面状态全景

| 状态 | 触发条件 | 页面表现 |
| :--- | :--- | :--- |
| 初始 | AC 清单存在 | 展示门禁 |
| 输入 | 筛选 AC | 更新列表 |
| 成功 | 全部 P0 通过 | 显示可进入 TDD |
| 失败 | P0 未通过 | 显示阻塞 |
| 异常 | 缺少契约 | 引导回总览 |
| 空状态 | 尚未生成验收契约 | 显示创建说明 |

### 7.2 边界与容错

- 网络异常：无需网络。
- 会话过期：无登录态。
- 连续错误：聚合未通过项。
- 异常输入：筛选无结果时回到全部。
- 并发操作：多人更新验收时以 Git 差异为准。

## 8. 业务规则

- P0 未通过时不得判定完成。
- 缺源码引用的结论不得作为最终结论。
- 脚本越界即验收失败。

## 9. 页面流转

```text
验收契约页
  ├─ 点击总览 → 总览页
  └─ 点击进入 TDD master → 后续链路
```

## 11. 验收标准

### 功能验收

- 展示 P0/P1/P2 AC。
- 展示通过和未通过数量。
- 展示未验证结论。
- 能跳转总览。
- 能指向 TDD 下一步。

### 体验验收

- 完成定义清晰。
- 阻塞项明显。
- AC 可复制。
- 筛选可用。
- 移动端可读。

### 安全验收

- 不绕过 P0 门禁。
- 不隐藏未验证结论。
- 不自动修改源码。
- 不展示敏感信息。
- Git 状态作为最终确认。
