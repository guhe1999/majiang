# Mahjong Project Parse 技术方案

## 1. 产品技术画像

Mahjong Project Parse 是一个内部源码解析与协作交付工作台。MVP 目标不是上线多人 Web 服务，而是把 `code-back/mahjong` 解析成文档、页面原型、轻量索引和验收契约。

### 1.1 产品形态

- 产品类型：静态 Web 原型 + Markdown 文档体系。
- 主要入口：浏览器本地打开 HTML。
- 端的数量：单端，桌面 Web 为主，移动端可读响应式。

### 1.2 核心技术约束

| 约束 | 来源（PRD 章节） | 技术含义 |
| :--- | :--- | :--- |
| 第一版不改业务代码 | §0 / §7 | 脚本默认只读源码，不引入自动重构 |
| 文档完整性和源码引用是门禁 | §10 | 需要 Markdown 与索引检查 |
| 本地可读即可 | 设计蓝图 §8 | 不需要服务端和数据库 |
| 国内可达 | design-master 铁律 | 不依赖海外托管服务 |

### 1.3 团队与规模画像

- 团队规模假设：2 人协作 + Coding Agent。
- 技术栈偏好：轻量、本地、可读、可版本管理。
- 用户规模预期：PoC / 内部早期。
- 上线时间压力：中，优先完成可交付文档链路。

### 1.4 非功能性需求

- 性能：本地静态页面秒级打开。
- 可用性：无网络也可阅读。
- 安全：不复制敏感数据，不自动修改业务代码。
- 可审计：所有文档和索引通过 Git 追踪。

## 2. 技术栈总览（一张图）

```text
前端层    原生 HTML/CSS/JavaScript + Tailwind CDN + Phosphor Icons
文档层    Markdown + 页面文档 + PRD / 设计 / TDD 链路产物
数据层    本地 JSON 索引 + Markdown 文件
脚本层    Shell 扫描脚本 + 同步检查脚本
基础设施  本地文件系统 + Git 版本管理
```

## 3. 前端技术选型

### 3.1 候选方案对比

| 候选 | DX | Perf | Eco | AI | Scale | Learn | TCO | 加权总分 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| 原生 HTML/CSS/JavaScript + Tailwind CDN | 5 | 5 | 5 | 5 | 2 | 5 | 5 | 4.6 |
| React 19.2.7 + Vite 8.1.4 | 4 | 4 | 5 | 5 | 4 | 4 | 3 | 4.1 |
| Astro 7.0.7 静态站点 | 4 | 5 | 4 | 4 | 4 | 3 | 4 | 4.0 |

### 3.2 推荐：原生 HTML/CSS/JavaScript + Tailwind CDN

**推荐理由**：
1. MVP 只需要静态原型和文档展示，不需要构建链。
2. 本地打开即可使用，符合无数据库、无后端、无部署的约束。
3. AI 生成和维护原生 HTML 的可靠性最高。
4. Tailwind CSS 版本已通过 `npm view tailwindcss version` 验证为 4.3.2。

**核心配置建议**：
- 路由方案：同目录静态 HTML 互链。
- 状态管理：页面内少量 JavaScript。
- 样式方案：tokens.css + 页面内 CSS 变量引用。
- 图标：Phosphor Icons 2.1.2。

**次选**：Astro 7.0.7。若后续文档量增大并需要自动生成静态站点，再切换。

**不推荐**：React/Vite。当前阶段引入构建链会增加复杂度，收益不足。

### 3.3 风险与缓解

- 风险：HTML 文件重复样式多。缓解：阶段二再抽公共模板。
- 风险：CDN 字体或图标不可达。缓解：核心内容不依赖图标，必要时本地化资源。

## 4. 后端技术选型

### 4.1 候选方案对比

| 候选 | DX | Perf | Eco | AI | Scale | Learn | TCO | 加权总分 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| 无后端服务 | 5 | 5 | 5 | 5 | 1 | 5 | 5 | 4.7 |
| Go CLI 辅助工具 | 3 | 5 | 4 | 4 | 3 | 3 | 4 | 3.7 |
| Node.js 静态索引生成器 | 4 | 4 | 5 | 5 | 3 | 4 | 3 | 4.0 |

### 4.2 推荐：无后端服务

**推荐理由**：
1. PRD 明确第一版不做业务前端和复杂自动化。
2. 所有数据都能由 Markdown 和 JSON 索引承载。
3. 无后端服务降低运维和安全面。

**次选**：Node.js 静态索引生成器。若后续需要更复杂的索引生成，可引入。

**不推荐**：Go Web 服务。当前没有在线服务需求，引入服务是过度设计。

### 4.3 风险与缓解

- 风险：后续多人协作需要在线状态。缓解：阶段二再评估后端服务。
- 风险：索引生成逻辑散落在脚本中。缓解：保持脚本文档和输入输出清晰。

## 5. 数据层选型

### 5.1 主数据

| 候选 | DX | Perf | Eco | AI | Scale | Learn | TCO | 加权总分 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| Markdown + JSON 索引 | 5 | 5 | 5 | 5 | 2 | 5 | 5 | 4.6 |
| SQLite 本地库 | 4 | 5 | 5 | 4 | 3 | 4 | 4 | 4.1 |
| PostgreSQL 自托管 | 2 | 5 | 5 | 5 | 5 | 3 | 2 | 3.7 |

**推荐**：Markdown + JSON 索引。

**推荐理由**：
1. 文档是主资产，Markdown 最适合 Git 追踪。
2. JSON 索引能支撑静态页面搜索和状态展示。
3. 数据规模小，不需要数据库。

**次选**：SQLite。若后续需要本地状态持久化和查询，可升级。

**不推荐**：PostgreSQL。MVP 数据量太小，引入数据库过度设计。

### 5.2 缓存层

不激活。静态文件和浏览器缓存已足够。

### 5.3 搜索引擎

不激活。MVP 通过 JSON 索引和前端筛选即可。

## 6. AI 能力选型

本阶段不激活 AI/LLM 能力。PRD 目标是解析和文档化，AI 能力由 Codex 当前链路承担，不需要在产品技术栈中接入模型、向量库或 Agent 框架。

## 7. 基础设施选型

### 7.1 部署与托管

| 候选 | DX | Perf | Eco | AI | Scale | Learn | TCO | 加权总分 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| 本地文件系统 + Git | 5 | 5 | 5 | 5 | 2 | 5 | 5 | 4.7 |
| 阿里云 OSS + CDN | 3 | 5 | 4 | 4 | 4 | 3 | 3 | 3.8 |
| Docker + 国内云 ECS | 2 | 4 | 5 | 4 | 5 | 3 | 2 | 3.5 |

**推荐**：本地文件系统 + Git。

**次选**：阿里云 OSS + CDN。若需要给更多人访问静态页面，可上传静态产物。

**不推荐**：Docker + ECS。当前没有服务端，不需要容器。

### 7.2 认证与授权

**推荐**：无登录认证。第一版所有人看同一套内容。

**次选**：本地静态密码说明页。仅当需要临时分享时使用。

**不推荐**：第三方认证服务。当前没有账号体系需求。

### 7.3 可观测性

**推荐**：Git 差异 + 脚本日志 + 文档同步检查。

**次选**：本地日志文件。若脚本数量增加，可写入 `script/logs/`。

**不推荐**：SaaS 监控。当前无线上服务，不需要。

### 7.4 CI/CD

推荐：暂不接 CI。阶段二如需要，可用本地脚本或国内 Git 平台流水线检查文档同步。

### 7.5 条件能力

- 实时通信：不激活。
- 文件/富媒体：不激活。
- 支付/计费：不激活。
- 多端/跨平台：不激活。

## 8. 实施建议与风险全景

### 8.1 推荐技术栈一览（最终决断）

| 层级 | 推荐技术 | 版本 / 备注 |
| :--- | :--- | :--- |
| 前端原型 | 原生 HTML/CSS/JavaScript | 无构建链 |
| 样式辅助 | Tailwind CSS | 4.3.2，已通过 npm 验证 |
| 图标 | Phosphor Icons | 2.1.2，已通过 npm 验证 |
| 文档 | Markdown | Git 追踪 |
| 数据索引 | JSON | 本地静态索引 |
| 脚本 | Shell | 文件扫描和同步检查 |
| 部署 | 本地文件系统 + Git | 无网络依赖 |

### 8.2 第一周脚手架建议（MVP 范围）

1. 固定 `output/` 与 `document/` 的同步规则。
2. 补齐 6 个页面文档。
3. 生成静态 HTML 原型。
4. 创建脚本索引清单。
5. 创建源码扫描脚本。
6. 创建文档同步检查脚本。
7. 将最终项目文档同步到 `document/`。
8. 用验收契约走查 P0 AC。

### 8.3 后续阶段扩展预留

| 扩展点 | 为哪个阶段/能力预留 | 预留方式 |
| :--- | :--- | :--- |
| 静态索引生成器 | 阶段二：更复杂文档索引 | Shell 可升级为 Node.js 或 Go CLI |
| 本地数据库 | 阶段二：状态持久化 | JSON 可升级 SQLite |
| 在线工作台 | 阶段三：多人协作 | 静态页面可迁移到 Web App |
| 测试门禁 | TDD master | 验收契约已提供入口 |

### 8.4 全局风险与缓解

| 风险 | 影响 | 缓解措施 |
| :--- | :--- | :--- |
| 静态页面重复代码多 | 后续维护成本增加 | 阶段二再抽模板或静态生成器 |
| JSON 索引手工维护易过期 | 页面状态不准 | 增加同步检查脚本 |
| 文档结论缺源码引用 | 影响可信度 | 验收契约中将其标记为 P0 门禁 |

### 8.5 技术负责人的最终结论

#### 是否可以立即开工

- 明确选择：可以开工。

#### 最大的技术不确定性

> 源码真实职责需要只读解析后确认，不能只依赖文件名推断。

#### 现在最不应该做的事

> 不应该引入数据库、后端服务、登录权限或复杂自动化。MVP 的价值是把知识结构和验收契约先建立起来。
