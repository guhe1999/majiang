# Mahjong Project Parse 摘要版

**版本**: v1
**完整 PRD**: output/PRD详细版.md
**开发版**: output/PRD-dev.md
**阅读时长**: 5 分钟

## 1. 这是什么

把 `code-back/mahjong` 整理成可学习、可协作、可验收的项目知识包。

## 2. 为什么做

已有麻将后端源码，但缺少系统文档和验收契约，直接改代码会放大协作风险。

## 3. 给谁做

- **决策者**: 当前协作团队。
- **使用者**: lei、he、后续 Coding Agent。
- **不是谁**: 不是外部玩家，不是线上运营用户。

## 4. 怎么解决

1. 先把项目结构、模块职责和核心流程解析清楚。
2. 再沉淀文档、影响范围表、迁移计划和验收契约。
3. 最后让 design-master、tdd-master 和 Coding Agent 继续承接。

## 5. 第一步先做什么

- **MVP 先做**: 全套解析文档、迁移计划、轻量脚本和验收契约。
- **做完用户能**: lei 能讲清源码，he 能判断需求影响范围。
- **要验证**: 文档是否足以支持后续需求和开发。
- **MVP 完成定义**: 完成 F1、F2、F3、F4、F5、F6、F7 即视为完成。
- **先不做**: 不改业务代码，不重构核心逻辑，不直接做前端。

## 6. 关键功能

| # | 功能 | 所属阶段 | 解决什么 |
|---|---|---|---|
| 1 | 项目画像 | MVP | 让团队知道项目是什么 |
| 2 | 模块地图 | MVP | 让 lei 能讲清结构 |
| 3 | 核心流程 | MVP | 让团队理解麻将后端如何运转 |
| 4 | 影响范围表 | MVP | 让 he 能判断新需求影响 |
| 5 | 验收契约 | MVP | 让后续交付有门禁 |

## 7. 成功长什么样

| 指标 | 当前 | 第一版目标 |
|---|---|---|
| 源码结构可讲清 | 初步目录清单 | lei 能按文档讲清入口、模块和流程 |
| 需求影响可判断 | 依赖经验判断 | he 能按表判断影响范围 |
| 交付可验收 | 无门禁 | 有验收契约 |

## 8. 资源 + 时间

- 团队配置: lei + he + Codex 协作。
- 关键依赖: 本地源码、目录约定、链路产物。
- 预计上线: 第一版文档链路完成后即可进入设计和 TDD。

## 9. 最大风险

1. 源码理解误判，缓解方式是所有结论回指源码。
2. 文档目录混乱，缓解方式是 `output/` 和 `document/` 分工明确。
3. 验收契约过泛，缓解方式是用 TDD master 生成门禁。

## 10. 需要拍板的事

当前无阻塞 PRD 的开放问题。

## 11. 完整信息怎么看

- `output/PRD详细版.md`：完整 PRD。
- `output/PRD-dev.md`：开发版。
- `assumptions.md`：假设清单。
- `debate-log.md`：决策记录。
- `evidence/`：本地依据说明。
