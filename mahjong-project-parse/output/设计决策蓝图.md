# 设计决策蓝图（已与用户逐轮确认）

## 1 盘子

- 端形态：桌面 Web 工作台为主。
- 响应式：移动端只做可读响应式，不做完整移动端工作流。
- 工作台聚合：第一屏聚合项目解析进度、源码概览、关键风险、待验收项、最近文档入口。

## 2 信息架构 & 站点地图

- 导航模式：左侧导航工作台。
- 一级分区：
  - 总览
  - 模块地图
  - 核心流程
  - 影响范围
  - 脚本索引
  - 验收契约
- 页面调用关系骨架：
  - 总览 → 模块地图 / 核心流程 / 验收契约
  - 模块地图 → 核心流程 / 影响范围
  - 核心流程 → 影响范围 / 验收契约
  - 影响范围 → 验收契约
  - 脚本索引 → 总览 / 验收契约

## 3 角色 & 权限

- 账号体系：第一版不做登录。
- 权限策略：不区分真实权限，所有人看到同一套内容。
- 角色处理：提供角色视角筛选或标记，帮助读者理解 lei / he / Coding Agent 分别最关心什么。
- 同页角色差异：不隐藏、不置灰，只用标签、筛选或推荐阅读顺序区分。

## 4 各功能页面拆解

- 总览页
  - 页面骨架：项目状态、源码概览、解析进度、关键风险、最近文档、下一步入口。
  - 空状态：提示尚未生成项目画像，提供“查看 PRD / 运行扫描脚本”入口。
  - 加载状态：显示文档索引和源码统计加载中。
  - 错误状态：提示索引缺失或扫描失败，给出脚本索引页入口。
- 模块地图页
  - 页面骨架：模块树、文件族分组、职责摘要、依赖关系、源码引用。
  - 空状态：提示尚未解析模块，展示 `code-back/mahjong` 文件清单入口。
  - 加载状态：显示模块树骨架屏。
  - 错误状态：提示源码引用缺失或模块分类待确认。
- 核心流程页
  - 页面骨架：流程列表、流程图、涉及模块、关键状态、异常路径、源码引用。
  - 空状态：提示尚未完成核心流程解析。
  - 加载状态：显示流程图和流程列表加载中。
  - 错误状态：提示流程结论缺少源码引用。
- 影响范围页
  - 页面骨架：需求类型表、影响模块、影响流程、风险等级、验收点、推荐阅读。
  - 空状态：提示尚无需求样例，可从 PRD 模拟需求开始。
  - 加载状态：显示影响矩阵加载中。
  - 错误状态：提示需求无法归类，建议标记开放问题。
- 脚本索引页
  - 页面骨架：脚本清单、用途、输入输出、安全边界、最近运行结果。
  - 空状态：提示尚未添加轻量脚本。
  - 加载状态：显示脚本运行状态加载中。
  - 错误状态：提示脚本失败、超时或越界风险。
- 验收契约页
  - 页面骨架：完成定义、AC 清单、门禁状态、未验证结论、下一步 TDD 入口。
  - 空状态：提示尚未生成验收契约。
  - 加载状态：显示门禁清单加载中。
  - 错误状态：提示缺文档、缺源码引用或缺影响范围表。

## 5 通用交互范式

- 全站交互：文档工作台交互。
- 搜索：全局搜索文档标题、源码引用、模块名、流程名、验收项。
- 筛选：按页面、文档类型、风险等级、是否有源码引用、是否已验收筛选。
- 锚点目录：长文档页面右侧提供本页目录和当前位置。
- 源码引用跳转：源码路径以可点击引用形式呈现，支持复制路径。
- 状态标签：使用“已验证 / 未验证 / 待补充 / 高风险 / 可进入 TDD”等标签。
- 列表：优先紧凑表格 + 分组，不做复杂批量操作。
- 表单：第一版不提供复杂编辑表单，只保留筛选、搜索和轻量备注入口。
- 反馈：Toast 用于复制路径、筛选保存；明显错误使用内联错误块。

## 6 视觉风格锚点

- 明暗模式：浅色为主。
- 风格定调：专业文档工具风，克制、清晰、可长时间阅读。
- 设计 Category：内部工程知识工作台。
- 情绪基调：可靠、清醒、低噪音、有轻微代码感。
- 交互哲学：色彩响应，用状态色表达风险、完成度和验证状态；不使用夸张动效。
- 字体气质：正文优先系统无衬线，代码和路径使用等宽字体。
- 设计签名：每个关键结论旁都有“源码引用 / 验收状态 / 影响范围”三联信息，形成产品识别点。

## 7 数据与集成

- 数据源：本地 Markdown / JSON 索引。
- 外部系统：第一版不接数据库、不接后端 API、不接外部服务。
- 核心数据实体：ProjectProfile、ModuleMap、CoreFlow、ImpactMatrix、ScriptIndex、AcceptanceContract。
- 数据生成方式：由文档、脚本扫描结果和人工解析结果生成静态索引。
- 数据同步：手动或脚本触发同步，不做实时同步。
- 数据读写：页面原型仅读取静态数据；真实文档更新仍以 Markdown 文件为准。

## 8 技术栈 & 架构 & 部署

- 技术路线：静态 HTML 原型 + Markdown 文档 + Shell 脚本。
- 前端：原生 HTML/CSS/JavaScript，避免引入构建链。
- 后端：第一版无后端服务。
- 存储：Markdown 文件和本地 JSON 索引。
- 脚本：Shell 优先，用于文件扫描、索引生成、文档同步检查。
- 部署：本地打开即可使用；可放在仓库中随文档一起版本管理。
- 国内可达：无外部托管依赖，无网络依赖。
- 架构原则：锚定 MVP，不为多人协作、登录权限、数据库和实时服务过度设计。

## 9 安全 & 运维

- 认证：第一版不做登录认证。
- 敏感数据：不复制真实用户数据；源码引用只记录路径、模块和结论，不粘贴大段源码。
- 业务代码保护：脚本默认只读 `code-back/mahjong`，禁止自动修改业务代码。
- 备份：依赖 Git 版本管理。
- 审计：关键决策记录在 `conversation.md`、`state.json` 和 `command/`。
- 危险操作可逆：不提供删除源码、重构源码、批量写业务代码等操作。
- 运维检查：通过脚本检查文档同步状态、索引存在性和源码引用缺失。
