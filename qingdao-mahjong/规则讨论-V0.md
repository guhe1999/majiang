# 青岛推倒胡规则讨论 V0

**状态**：调研草案，尚未确认  
**讨论范围**：首个麻将玩法的牌局规则和结算口径；不讨论登录、商城、支付、赛事和客户端实现。  
**结论纪律**：以下“候选”均不是正式规则。只有来源、产品负责人和验收口径三者一致后，才能进入 `output/PRD详细版.md`。

## 1. 首轮结论

公开资料显示，“推倒胡”是一个跨地域的玩法名称，并不存在可直接套用的唯一规则集。青岛本地麻将资料则呈现出“基础规则 + 房间选项”的组合：可带混、可选是否吃牌、可选一炮多响、平度 8 张、258 将等。[青岛麻将规则页](https://www.hzbianqu.com/bqwl/index5.html) 与 [青岛麻将介绍](https://www.sohu.com/a/335920176_120099887) 都支持这一判断。

另一方面，通用/北方推倒胡资料通常把“不允许吃牌”作为核心特征，但该资料明确介绍的是常州或泛北方玩法，不能证明它就是青岛规则。[游戏茶苑的推倒胡规则](https://www.gametea.com/news/201905/8712.html) 因此只可作为“推倒胡候选特征”的参考。

> 当前最重要的产品决策不是番数，而是：首版要交付“青岛麻将的可配置房间”，还是“固定为不让吃牌的青岛推倒胡房间”。两者的操作时序、听牌提示、胡牌校验和测试集均不同。

## 3. 两套候选玩法包

| 维度 | A. 青岛地方规则包（R-001/R-002） | B. 不让吃牌的推倒胡房（R-003 + 青岛房间选项） | 当前建议 |
| --- | --- | --- | --- |
| 人数与基础牌 | 4 人、136 张、庄 14 / 闲 13 | 同左 | 作为首版共同候选基线，待确认。 |
| 行牌方向 | 资料描述为逆时针 | 通用资料未给出青岛方向 | 以 A 为候选，须用真实牌友/规则书确认。 |
| 吃牌 | 可吃上家牌；可有次数或“吃副打副手把一”限制 | 不允许吃牌 | 这是首个必须拍板项。 |
| 碰、杠 | 支持；明杠、暗杠、补杠及抢杠胡的具体结算需确认 | 支持；抢杠胡为常见候选 | 首版应分别写为动作合法性与结算规则，不能只用一个开关表达。 |
| 混牌 | 可选；混可替代胡牌用牌、不可当碰杠替代牌；有混时通常只许自摸；四混可直接结算等细节存在 | 不作为通用推倒胡必备项 | 若首版带混，必须先锁定翻混算法、混牌可否打出、持混可否点炮、四混结算。 |
| 胡牌路径 | 自摸、点炮，可选一炮多响；可有过胡限制 | 自摸、点炮、抢杠；有的产品可一炮多响 | 将“是否一炮多响”“过胡后何时解除”拆成独立规则。 |
| 特殊本地规则 | 扭牌、平度 8 张、258 将、扣听、荒庄翻倍、跑牌等 | 通常不含 | 全部后置为可选扩展，除非明确首版要服务对应本地桌规。 |
| 牌型与计分 | 屁胡、碰碰胡、七对/豪华七对、十三幺、天/地胡、清一色等；部分条件为乘数 | 大/小胡或倍数体系因地区而异 | 不能混用两套表；必须确定计分模型后才可写代码。 |

## 4. 候选共同基线（尚未确认）

若产品目标是先验证最小可玩闭环，可暂以以下内容作为讨论起点，而不是实现承诺：

1. 4 名玩家，使用万、筒、条、东南西北、中发白共 136 张；庄家 14 张、其余玩家各 13 张。
2. 每局按“发牌 → 轮流摸/打 → 响应操作 → 胡牌或荒庄 → 结算”推进。
3. 基础操作候选为打牌、碰、杠、胡；“吃”由第一个待决项决定。
4. 基础胡牌候选为标准 4 组面子 + 1 对将；七对、十三幺、十三不靠是否开放要逐项确认。
5. 基础结算候选为自摸三家支付、点炮由点炮者支付；一炮多响、杠分、抢杠胡、荒庄和连庄不得默认推断。

## 5. 首版房间的待决清单

下列项目任一项都会改变规则引擎、前端操作提示或测试契约；确认前不可进入实现。

| 优先级 | 待决项 | 候选值 | 影响 |
| --- | --- | --- | --- |
| P0 | 玩法定位 | A 地方青岛麻将 / B 不让吃牌的青岛推倒胡 | 决定是否生成吃牌机会和所有相关时序。 |
| P0 | 混牌 | 不带混 / 带混 | 改变牌墙、胡牌求解、可操作集和点炮资格。 |
| P0 | 胡牌方式 | 自摸 + 点炮 / 仅自摸 | 改变响应窗口和结算责任方。 |
| P0 | 一炮多响 | 开 / 关；若开，响应优先级与结算口径 | 改变状态机终局条件。 |
| P0 | 计分模型 | 固定分制 / 番数制；封顶和叠加规则 | 决定账本结算公式和回放展示。 |
| P1 | 基础牌型 | 仅标准胡 / 加七对、十三幺、十三不靠等 | 决定胡牌求解器和测试样例。 |
| P1 | 杠规则 | 明/暗/补杠、抢杠、杠上花及杠分责任 | 决定动作抢占和中途结算。 |
| P1 | 庄家与荒庄 | 庄家轮换、连庄、留牌数、荒庄是否翻倍 | 决定跨局状态与房间结算。 |
| P2 | 本地扩展 | 扭牌、平度 8 张、258 将、扣听、跑牌 | 增加规则组合数，建议在核心闭环稳定后再纳入。 |

## 6. 建议的 MVP 收敛顺序

1. 先由项目负责人选择上表 P0 的玩法定位：是否“不让吃牌”。
2. 再确定一个固定房间模板（不要在 MVP 同时开放所有地方选项）。
3. 固化“牌张 → 行牌 → 操作优先级 → 胡牌 → 结算 → 庄家变化”的可执行规则表。
4. 为每个规则写正反例牌面和结算样例，再进入 PRD、设计和验收阶段。
5. 通过对局回放或本地牌友复核规则表后，才考虑把稳定规则开放为房间配置。

## 7. 对现有代码的影响提示（不是规则结论）

现有 `code-back/mahjong` 已包含吃牌检测和处理、混牌类型、多个牌型及番数表。因此：

- 若首版确定“不让吃牌”，需要从服务端操作机会生成处禁止吃牌，而不是仅在客户端隐藏按钮；初步关注 `engine/room_run.go`、`engine/user_hd.go`、`engine/user_hd_handle.go`。
- 若确定“带混”，现有代码中的 `HUN` 与箭牌集合存在既有语义，必须逐项验证能否匹配确认后的翻混和结算规则，不能直接复用。
- 现有番数表含门前清、混一色、青龙等项目，和 R-001 的固定分/倍数表不一致；首版结算要作为独立规则模块重建验收样例。

这些仅是影响范围初查，未构成代码改造方案。
